## Iteration 4 - US-004: Create regex-parser.ts for new RegExp() syntax
- Created `src/yaml/types/regex-parser.ts` with:
  - `parseRegex()` function to convert `new RegExp("pattern", "flags")` syntax to real RegExp objects
  - `isRegexString()` type guard to detect regex syntax strings
  - `RegexParsed` TypeScript interface
  - Support for patterns with or without flags
  - Support for double quotes, single quotes, and backticks
  - Invalid regex detection (catches constructor errors for invalid patterns/flags)
- Created `src/yaml/types/__tests__/regex-parser.test.ts` with 23 unit tests
- Files changed:
  - `src/yaml/types/regex-parser.ts` (new)
  - `src/yaml/types/__tests__/regex-parser.test.ts` (new)
- Learnings for future iterations:
  - RegExp constructor throws for invalid patterns/flags - wrap in try/catch
  - JavaScript string escaping differs from YAML escaping - in tests, `'\\d+'` = single backslash
  - Tests in `__tests__/` are integration tests, not unit tests

## Iteration 3 - US-003: Create date-parser.ts for new Date() syntax
- Created `src/yaml/types/date-parser.ts` with:
  - `parseDate()` function to convert `new Date("date-string")` syntax to real Date objects
  - `isDateString()` type guard to detect date syntax strings
  - `DateParsed` TypeScript interface
  - Support for date-only format (`"2024-01-15"`) and ISO timestamps (`"2024-01-15T10:30:00Z"`)
  - Support for double quotes, single quotes, and backticks
  - Invalid date detection via `Number.isNaN(date.getTime())`
- Created `src/yaml/types/__tests__/date-parser.test.ts` with 18 unit tests
- Files changed:
  - `src/yaml/types/date-parser.ts` (new)
  - `src/yaml/types/__tests__/date-parser.test.ts` (new)
- Learnings for future iterations:
  - Date validation pattern: `Number.isNaN(date.getTime())` catches invalid dates
  - ISO timestamp tests should use UTC methods (getUTCFullYear, getUTCMonth, etc.)
  - JavaScript months are 0-indexed (January = 0)

## Iteration 2 - US-002: Create promise-parser.ts for Promise.resolve/reject syntax
- Created `src/yaml/types/promise-parser.ts` with:
  - `parsePromiseResolve()` function to convert `Promise.resolve(value)` syntax to resolved Promise
  - `parsePromiseReject()` function to convert `Promise.reject(error)` syntax to rejected Promise
  - `parsePromise()` unified parser for either resolve or reject
  - `parseValue()` helper to parse inner value (JSON, primitives, nested special objects)
  - `isPromiseResolveString()`, `isPromiseRejectString()`, `isPromiseString()` type guards
  - `PromiseResolveParsed` and `PromiseRejectParsed` TypeScript interfaces
  - Support for nested error parsing via import from error-parser.ts
- Created `src/yaml/types/__tests__/promise-parser.test.ts` with 30 unit tests
- Files changed:
  - `src/yaml/types/promise-parser.ts` (new)
  - `src/yaml/types/__tests__/promise-parser.test.ts` (new)
- Learnings for future iterations:
  - Use `/s` flag in regex patterns for multi-line matching (though not needed for simple cases)
  - Parse inner value by trimming and checking type patterns (quoted strings, numbers, JSON)
  - Convert unquoted object keys to quoted for JSON.parse compatibility
  - Use async/await with `await expect(promise).rejects.toThrow()` for testing rejected promises

## Iteration 1 - US-001: Create error-parser.ts for new Error() syntax
- Created `src/yaml/types/error-parser.ts` with:
  - `parseError()` function to convert `new Error("message")` syntax to real Error objects
  - `isErrorString()` type guard to detect error syntax strings
  - `ErrorParsed` TypeScript interface
  - Support for custom error names via `{ name: "CustomError" }` syntax
  - Support for double quotes, single quotes, and backticks
  - Support for empty messages
- Created `src/yaml/types/__tests__/error-parser.test.ts` with 17 unit tests
- Files changed:
  - `src/yaml/types/error-parser.ts` (new)
  - `src/yaml/types/__tests__/error-parser.test.ts` (new)
- Learnings for future iterations:
  - Regex pattern `/^new Error\(\s*(['"`])(.*?)\1\s*(?:,\s*(\{[^}]+\}))?\s*\)$/` works well
  - Use non-greedy `.*?` to allow empty strings in capture groups
  - Parse options by extracting with another regex for name property

## Iteration 5 - US-005: Create index.ts barrel export with unified parser
- Created `src/yaml/types/index.ts` with:
  - Named exports for all parser functions, type guards, and interfaces
  - `parseSpecialObject(value: unknown)` unified parser that tries each parser in order

  - Returns original value unchanged if no parser matches
- Created `src/yaml/types/__tests__/index.test.ts` with 43 unit tests:
  - Tests for all 4 special object types
  - Tests for regular values passing through unchanged
  - Tests verifying all barrel exports are available
- Files changed:
  - `src/yaml/types/index.ts` (new)
  - `src/yaml/types/__tests__/index.test.ts` (new)
- Learnings for future iterations:
  - Barrel exports should use explicit named exports for clarity
  - Import parsers with `.js` extension for ESM compatibility
  - Unified parser order matters: Error before Promise (Promise can contain Error)
